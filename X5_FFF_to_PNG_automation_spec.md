# X5 `.fff` / 3F 到最终单张 PNG 的自动化工具开发说明

## 1. 项目目标

开发一个**从 Hasselblad Flextight X5 扫描得到的 `.fff` 文件出发，自动生成最终单张 `.png` 成片**的工具。

这个工具的目标不是简单“转格式”，而是把原本分散在扫描、反相、基础调色、裁切、清洁、输出这些步骤里的操作，整理成一条**可重复、可记录、可调参数的自动化流水线**。

---

## 2. 术语约定

- **`.fff` / 3F**：Hasselblad Flextight / FlexColor 工作流中的原始扫描文件
- **工作 TIFF**：从 `.fff` 渲染出来的 16-bit TIFF，作为后续图像处理的中间文件
- **最终 PNG**：面向查看、归档展示或进一步排版使用的单张成片
- **Preset**：一组固定的处理参数，用于让不同底片在相似条件下得到稳定输出
- **Pipeline**：从输入到输出的整个自动化链路

---

## 3. 工具边界

### 3.1 这个工具要做的事

1. 接收 `.fff` 文件
2. 将 `.fff` 转成稳定可处理的 16-bit TIFF
3. 对 TIFF 做负片处理所需的自动步骤
4. 进行基础调色与画面整理
5. 输出单张 PNG
6. 保存日志、参数和中间结果，保证可回溯

### 3.2 首版不做的事

1. 不做复杂 GUI，优先 CLI
2. 不做过度依赖 AI 的“自动审美判断”
3. 不在第一阶段做高难度局部精修
4. 不把首版建立在“纯手写解析 `.fff` 私有格式”的高风险路线之上
5. 不追求一开始就替代 Photoshop 的全部能力

### 3.3 设计原则

1. **先稳定，再智能**
2. **先中间 TIFF 管线，再谈 `.fff` 深度支持**
3. **先全局处理，再局部修图**
4. **先可重复，再追求风格化**
5. **所有自动步骤都必须可关闭、可覆盖、可记录**

---

## 4. 推荐产品形态

首版建议做成：

- 一个 **CLI 命令行工具**
- 配一个 `config.yaml`
- 支持单张处理与批处理
- 输出中间文件、最终文件和日志

后续再补：

- 简单预览界面
- 参数面板
- 接触表 / 对比视图
- 人工复核模式

---

## 5. 总体处理顺序

建议把整条流水线拆成下面 8 个顺序明确的步骤：

1. **输入与校验**
2. **FFF 转 16-bit TIFF**
3. **负片反相与片基校正**
4. **基础曝光 / 白平衡 / 密度整理**
5. **构图整理：旋转、裁切、边缘处理**
6. **画质整理：灰尘、轻度降噪、轻度锐化**
7. **风格层：预设色调、对比、统一输出风格**
8. **输出 PNG + 侧车日志**

这里最关键的是：  
**不要一上来就直接从 `.fff` 到“最终风格化 PNG”一把梭。**  
必须先建立一个可靠的“中间工作 TIFF”层，否则后面所有调试都会混乱。

## 6. 功能模块与先后顺序

## 6.1 模块 A：项目输入与任务管理

### 目标
负责接收文件、创建输出目录、读取配置、写日志。

### 输入
- 单个 `.fff`
- 文件夹中的多个 `.fff`
- 可选：已经渲染好的 16-bit TIFF（供调试）

### 输出
- 标准化任务目录
- 处理日志
- 参数快照

### 必要功能
- 文件存在性检查
- 扩展名检查
- 输出目录自动创建
- 每张图生成唯一任务 ID
- 保存本次使用的 preset / config

### 为什么先做
这是整个工具的骨架。没有这层，后面每个模块都像散兵游勇。

---

## 6.2 模块 B：FFF 适配层 / 渲染层

### 目标
把 `.fff` 稳定转换为后续处理所需的 **16-bit TIFF**。

### 建议策略
不要在第一阶段让 Codex 直接手写 `.fff` 深度解析器。  
更现实的做法是把这层设计成一个**可替换 backend**：

#### Backend 方案建议
- **Backend A（优先）**：调用已有稳定软件或既有导出流程，把 `.fff` 转为 16-bit TIFF
- **Backend B（开发调试）**：允许直接输入 16-bit TIFF，跳过 `.fff`
- **Backend C（未来）**：如果后面验证可行，再研究 `.fff` 更深层的直接读取能力

### 必要功能
- 接收 `.fff`
- 生成中间 TIFF
- 保留原始分辨率与位深
- 输出元数据 JSON
- 检测转换失败并报错

### 为什么第二个做
`.fff` 是入口。如果这个入口不稳，后面做得再漂亮也只是空中楼阁。

---

## 6.3 模块 C：负片反相与片基校正

### 目标
把工作 TIFF 从“扫描状态”转成“可读图像状态”。

### 这里要解决的问题
- 负片反相
- 橙色片基 / 底色基线处理
- 各通道初始平衡
- 黑白负片与彩色负片分流

### 建议拆成两个子步骤
1. **Film Base Estimation**
   - 自动采样边缘未曝光区域
   - 估计片基颜色
2. **Negative Inversion**
   - 基于片基估计做通道反相与归一化

### 必要功能
- 自动识别黑白 / 彩负
- 支持手动覆盖片型
- 支持用户指定片基采样区域
- 输出“反相后 TIFF”供检查

### 为什么放在这里
这是从“材料”进入“图像”的那一刻。  
如果这一步不稳，后面的调色本质上都在给错误基础打补丁。

---

## 6.4 模块 D：基础调色层

### 目标
完成“可用图像”的基础整理，而不是最终美学风格。

### 处理内容
- 自动曝光校正
- 白平衡 / 中性色归正
- 高光阴影压缩
- 全局对比控制
- 基础色偏修正

### 必要功能
- 自动模式
- 预设模式
- 参数覆盖模式
- 输出前后对比图

### 建议原则
首版只做**全局调整**，不要急着做局部 dodge & burn。

### 为什么这一阶段重要
很多自动化工具失败，不是因为算法不强，而是因为它一开始就试图做“最终审美”。  
首版应该只负责把图像带到一个**干净、稳定、可继续工作**的状态。

---

## 6.5 模块 E：构图整理层

### 目标
自动处理扫描后常见的几何与画面边界问题。

### 处理内容
- 自动旋转校正
- 自动裁切
- 黑边 / 白边 / 底片边缘检测
- 保留边框模式与去边框模式
- 纵横比策略

### 模式建议
- `crop_mode: clean`
- `crop_mode: keep_border`
- `crop_mode: custom_ratio`

### 必要功能
- 自动检测边缘
- 允许用户关闭自动裁切
- 支持固定比例输出
- 保存裁切框数据

### 为什么这一步单独做
构图不是颜色问题，颜色也不是构图问题。  
把它拆开，后期 debug 会容易很多。

---

## 6.6 模块 F：画质整理层

### 目标
做轻量但必要的技术清洁，而不是修得“没有生命”。

### 处理内容
- 灰尘点检测
- 小面积污点修补
- 轻度降噪
- 输出前锐化
- 可选颗粒保护

### 建议原则
- 默认非常克制
- 强修复必须可关闭
- 先做“技术性清洁”，不做“皮肤式美容”

### 必要功能
- 强度分级
- 灰尘修补前后 diff 图
- 局部失败时可跳过

### 注意
这一层建议放到中后期再做。因为它最容易引入“看起来很干净，但其实很假”的副作用。

---

## 6.7 模块 G：风格预设层

### 目标
在技术正确之后，再决定成片的气质。

### 处理内容
- 对比曲线 preset
- 饱和度策略
- 色彩倾向 preset
- 输出 gamma / viewing intent
- 针对不同胶卷建立 preset

### 预设例子
- `neutral_archive`
- `soft_portra_like`
- `contrast_bw_clean`
- `warm_print_ready`

### 必要功能
- preset 可继承
- preset 可局部覆盖
- 输出时记录所用 preset 名称

### 为什么放这么后
“风格”不能替代“正确”。  
先把图像扶正，再谈语言。

---

## 6.8 模块 H：输出与归档层

### 目标
把处理结果以明确、稳定、可追踪的形式落盘。

### 输出内容
- 最终 PNG
- 中间 TIFF
- 低分辨率预览图
- sidecar JSON
- 日志文件
- 可选接触表

### 推荐目录结构

```text
project/
  input/
    xxx.fff
  output/
    xxx/
      01_raw_meta/
      02_work_tiff/
      03_inverted/
      04_base_grade/
      05_crop/
      06_cleanup/
      07_final/
        xxx_final.png
        xxx_final_preview.jpg
        xxx_sidecar.json
        xxx_log.txt
```

### sidecar JSON 至少记录
- 输入文件名
- 处理时间
- 处理阶段耗时
- 使用的 preset
- 自动裁切框
- 曝光 / 白平衡参数
- 锐化 / 降噪参数
- 失败 / 警告信息

## 7. 推荐开发阶段

## 阶段 0：需求冻结与样本准备

### 目标
在写代码前把问题边界钉住。

### 必做事项
1. 准备 10 到 30 张真实 `.fff` 样本
2. 样本覆盖：
   - 彩色负片
   - 黑白负片
   - 曝光正常
   - 欠曝 / 过曝
   - 有边框 / 无边框
   - 灰尘较多 / 较少
3. 明确“最终 PNG”的标准：
   - 是否保留底片边框
   - 色彩倾向偏中性还是偏风格化
   - 输出尺寸是否固定
4. 确定 MVP 不支持什么

### 产出
- 样本数据集
- 成片参考集
- 需求冻结文档

---

## 阶段 1：最小可运行骨架

### 目标
先让工具“跑起来”。

### 范围
- CLI
- 读取配置
- 输入输出目录
- 日志
- 支持直接喂 16-bit TIFF

### 成功标准
运行命令后能稳定生成任务目录与日志，不报结构性错误。

### 这一阶段不要做
- 自动调色
- 智能裁切
- 灰尘修补

---

## 阶段 2：FFF 转 TIFF

### 目标
打通真正的入口。

### 范围
- `.fff` 输入
- backend 调用
- 中间 TIFF 输出
- 转换异常处理

### 成功标准
至少 80% 的测试 `.fff` 能稳定转成 16-bit TIFF，并被后续模块读取。

### 风险点
这是全项目最大的不确定性。  
因此必须允许“跳过 `.fff`，直接从 TIFF 开始”作为备用开发路径。

---

## 阶段 3：基础图像可读化

### 目标
让扫描结果变成“能看”的图。

### 范围
- 反相
- 片基校正
- 基础曝光
- 白平衡
- 全局对比

### 成功标准
输出一张“技术上正确、观感上基本成立”的中间结果图。

### 这阶段的判断标准
不是“惊艳”，而是：
- 不偏得离谱
- 不灰得发闷
- 不炸高光
- 不脏色泛滥

---

## 阶段 4：构图与边缘整理

### 目标
让图像从“能看”变成“能交付”。

### 范围
- 自动旋转
- 自动裁切
- 黑边 / 白边处理
- 保边框模式

### 成功标准
大多数样本能得到合理裁切，不出现大面积误裁。

### 注意
这一阶段要保留“人工覆盖参数”，否则一旦误裁就没法救。

---

## 阶段 5：画质整理

### 目标
补上最基本的技术修饰。

### 范围
- 轻度灰尘处理
- 轻度降噪
- 轻度锐化
- 可选颗粒保护

### 成功标准
在 100% 放大下，瑕疵减少，但纹理没有被抹平。

---

## 阶段 6：预设与最终输出

### 目标
形成真正能长期使用的自动化工具。

### 范围
- 预设系统
- final PNG
- preview JPG
- sidecar JSON
- 批处理
- 接触表

### 成功标准
用户可以通过一条命令处理整批 `.fff`，并得到稳定、可回溯、风格一致的输出。

---

## 阶段 7：增强阶段（可选）

### 可以后做的增强项
- GUI 预览
- 人工复核界面
- 自动质量评分
- 多卷统一色彩
- 每卷自动建立参考基线
- AI 辅助灰尘检测
- Photoshop/外部软件联动导出

## 8. 推荐 MVP 范围

为了避免项目一开始过大，建议 MVP 只做下面这些：

1. 输入 `.fff` 或 16-bit TIFF
2. 得到中间 TIFF
3. 自动反相
4. 自动片基校正
5. 自动基础曝光 / 白平衡
6. 自动裁切
7. 输出最终 PNG
8. 保存 sidecar JSON 和日志

### 不放进 MVP 的内容
- 复杂 GUI
- 高级局部修补
- 智能审美评分
- 多图拼接
- AI 风格迁移
- 大规模胶卷库管理

## 9. 推荐技术路线

## 9.1 语言
优先 Python。

原因：
- 图像处理生态成熟
- 适合做 CLI
- 方便后续接 GUI
- 方便写 batch pipeline
- Codex 较容易稳定产出

## 9.2 工程结构建议

```text

  app/
    cli.py
    config.py
    runner.py
  adapters/
    fff_backend.py
    tiff_loader.py
  pipeline/
    invert.py
    film_base.py
    grade_basic.py
    crop.py
    cleanup.py
    export.py
  models/
    task.py
    result.py
    preset.py
  utils/
    io.py
    image_ops.py
    logging.py
    profiling.py
tests/
configs/
docs/
```

## 9.3 推荐依赖方向

- CLI：`typer` 或 `click`
- 配置：`pydantic` + `yaml`
- 数值计算：`numpy`
- 图像处理：`opencv-python`、`Pillow`、`tifffile`

- 测试：`pytest`

### 原则
不要让底层依赖过多。  
首版要克制，不要一开始就堆很多库。

## 12. 验收标准

## 12.1 技术验收
- 工具能稳定处理一批测试样本
- 不因单张失败导致整个批次中断
- 每张图都有日志和 sidecar
- 处理结果可复现

## 12.2 图像验收
- 颜色不严重跑偏
- 裁切正确率达标
- 高光阴影不过度损伤
- 不出现明显的自动修图痕迹
- 最终 PNG 可直接用于查看和进一步排版

## 12.3 工作流验收
- 用户不需要每张图手动重复一遍基础步骤
- 参数可以复用
- 中间结果可检查
- 失败时可回退、可继续、可改单个参数重跑

## 13. 给 Codex 的明确要求

你可以把下面这段直接放进任务里：

### 开发要求
1. 先做 **CLI + 配置 + 日志 + 中间文件目录结构**
2. 再做 **TIFF 处理主链路**
3. 把 `.fff` 支持设计成 **adapter / backend**，不要让整个系统和某一种 `.fff` 读取方式强耦合
4. 所有处理步骤必须支持：
   - 开关
   - 参数覆盖
   - 中间结果导出
5. 所有自动处理都要留下 sidecar JSON
6. 先确保稳定性，再逐步增加“更聪明”的自动化
7. 第一版重点是“从 `.fff` 到可交付 PNG 的可重复流程”，不是“做一个万能修图软件”

---

## 14. 最终建议

这个项目真正的难点，不在“PNG 导出”，也不在“写一些图像处理函数”。  
真正的难点在于两件事：

1. **把 `.fff` 入口做稳**
2. **把自动化限制在真正适合自动化的层面**

因此正确顺序应该是：

**骨架先行 → `.fff` 转 TIFF → 反相与基础整理 → 裁切 → 轻清洁 → 预设 → 输出**

而不是：

**一开始就追求一键得到“最好看”的成片**

前者是在建系统。  
后者很容易变成一堆彼此打架的魔法参数。

---

## 15. 一句话版项目定义

这是一个以 `.fff` / 3F 为输入、以 16-bit TIFF 为中间层、以最终单张 PNG 为输出的**可重复、可配置、可追踪**的负片自动化成片工具。
